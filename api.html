Introduction

Integrating with Safely Finance is quick and easy. Our docs provide the guidance your team needs to add Safely Finance to your checkout experience.

What is Safely Finance?

With Safely Finance you can add modern payment options to your checkout experience. Give your customers the options to pay with the convenience of Apple or Google Pay, or pay over time. No matter what they choose, Safely Finance has you covered.

Customers want payment options because moving is stressful. With Safely Finance, you can give customers options that make these costs flexible and convenient.

Data Types

Where applicable, the Safely Finance API uses a variety of standards for representing common pieces of data. More information can be found on each of these below.

Dates & Times

All dates are strings in the ISO 8601 format. The date representation is YYYY-MM-DD (for example, January 22, 2023 is represented as 2023-01-22). All datetimes are stored and returned in UTC (for example 2023-01-22T14:40:00Z).

Identifiers (IDs)

All IDs are strings represented by a Universally Unique IDentifier (UUID). The identifiers will follow a standard format (including hyphens) (for example fa2facc6-31a8-44e7-bac3-719fac7062ee).

Phone Numbers

All phone numbers are strings that follow the E.164 standard for international phone number formatting. The format is as follows,

[+][country code][area code][local phone number]

As an example, the US number (415) 555-0145 should be sent as the string +14155550145

Embedded Flow

The Safely Finance embedded flow helps you implement alternative payment options for your prospective and current customers within your existing website. The flow is initiated when a call is made to the Safely Finance API to create a Checkout.

You can see more details, along with screenshots, in the Safely Checkout overview.

Idempotent Requests

The Safely Finance API supports the use of an idempotency_key. This key is used for safely retrying requests without accidentally performing the same operation twice.

All POST requests accept idempotency keys.

Rate Limiting

The Safely Finance API limits the number of requests that can be received within a second. Limits are imposed to prevent abuse.

Certain high-traffic applications may qualify for an increase to rate limits. Please contact support to request an increase.

Response Codes

The Safely Finance API uses conventional HTTP response codes to indicate the success or failure of API requests.

Typically, codes that are in the 2xx range indicate success. Codes that fall within the 4xx range indicate an error contain a message body with a description of the error.

An example of an error response,

{
  "status_code": 400,
  "message": "The request body was missing required parameters",
  "code": "missing-params"
}

Security

Authentication

The Safely Finance API uses token authentication (also known as bearer authentication).

Consumers of the API can request a private token that can be used to make API requests. The token must be sent in the Authorization header when making all requests.

Failure to pass a token with your request, or providing an invalid token, will result in a 401 Unauthorized response following the format described in Response Codes.

Example Request

curl https://api.safelyfinance.co/hello \
  -X GET \
  -H "Authorization: Bearer sf_live_fkAk32AAFklj43KJKfk" \
  -H "Content-Type: application/json"

Headers

Any clients interfacing with the Safely Finance API must send the appropriate request headers.

Content-Type

All requests that have a request body (typically POST and PUT) must also pass a content-type header of application/json.

Accept

All requests must accept application/json, the standard response format for the Safely Finance API.

HTTPS

The Safely Finance API requires all communications to be secured using TLS 1.2 or greater.
